## 2. TÍNH NĂNG CÔNG VIỆC PHỤ THUỘC & PHÁT HIỆN CHU TRÌNH (TASK DEPENDENCIES & CYCLE DETECTION)

### A. Vấn đề chu trình phụ thuộc tuần hoàn gián tiếp
* **Vấn đề**: Hệ thống chỉ kiểm tra phụ thuộc tuần hoàn trực tiếp (A ↔ B) trong `addDependency` mà bỏ qua vòng lặp gián tiếp (A → B → C → A). Điều này phá vỡ luồng tính toán điều kiện hoàn thành công việc.

### B. Thuật toán phát hiện chu trình (DFS)
* **Backend**: Sử dụng thuật toán duyệt đồ thị theo chiều sâu (**DFS đệ quy**) trên bảng `TaskDependency` để tìm kiếm đường đi từ task tiên quyết (`dependsOnId`) đến task đích (`taskId`). Nếu tìm thấy, chặn thiết lập và trả về lỗi 400.
  ```typescript
  async function hasCircularPath(currentId: string, targetId: string, visited = new Set<string>()): Promise<boolean> {
    if (currentId === targetId) return true;
    visited.add(currentId);

    const dependencies = await prisma.taskDependency.findMany({
      where: { taskId: currentId },
      select: { dependsOnId: true },
    });

    for (const dep of dependencies) {
      if (!visited.has(dep.dependsOnId)) {
        const found = await hasCircularPath(dep.dependsOnId, targetId, visited);
        if (found) return true;
      }
    }
    return false;
  }
  ```
* **Frontend**: Triển khai thuật toán DFS tương tự (`isReachable`) trong [TaskDetailModal.tsx](file:///c:/project/task-flow/frontend/src/components/task/TaskDetailModal.tsx) để ẩn các task có khả năng tạo chu trình khỏi dropdown chọn.
* **Ràng buộc DONE**: Chặn chuyển trạng thái sang `DONE` tại `updateTaskStatus` và `reorderTasks` nếu bất kỳ task phụ thuộc tiên quyết nào chưa đạt trạng thái `DONE`.

## 4. DỒN DỊCH THỨ TỰ CÔNG VIỆC (TASK REORDERING GAPS & TRANSACTIONS)

* **Vấn đề**: Khi xóa một task hoặc thay đổi trạng thái qua Modal chi tiết, giá trị thứ tự `order` trong cột bị đứt gãy (lệch chỉ số hiển thị).
* **Giải pháp**: Sử dụng Prisma transaction để dồn dịch tự động:
  * **Khi xóa task**: Giảm `order` của các công việc đứng sau trong cùng cột đi 1:
    ```typescript
    prisma.task.updateMany({
      where: { projectId: task.projectId, status: task.status, order: { gt: task.order } },
      data: { order: { decrement: 1 } }
    })
    ```
  * **Khi đổi trạng thái qua Modal**: Thực hiện đồng thời giảm `order` các task đứng sau ở cột cũ, đếm số lượng task ở cột mới và gán `order` mới ở cuối cột mới cho task được chuyển đến.

## 5. VALIDATION ĐẦU VÀO & PHÒNG CHỐNG CRASH REQUESTS

* **Validation Schemas ([validation.ts](file:///c:/project/task-flow/backend/src/utils/validation.ts))**: Bổ sung `.trim()` làm sạch khoảng trắng thừa và `.max(...)` giới hạn kích thước chuỗi đầu vào của 14 schemas để bảo vệ bộ nhớ đệm và ngăn chặn lỗi vỡ CSS hiển thị ở Client.
* **Bảo vệ Query Parameters**: Thay thế ép kiểu TypeScript (`as string`) bằng việc kiểm tra kiểu thực tế `typeof === 'string'` trong các controller trước khi truyền tham số vào Prisma (phòng chống tấn công Query Parameter Array Injection gây crash API 500).

## 6. XÁC THỰC NGÀY THÁNG & THỜI GIAN (DATE & TIME VALIDATION)

### A. Chặn hạn chót (Deadline) trong quá khứ
* **Frontend (`TaskFormModal.tsx`)**:
  * Tích hợp `disabledDate={current => current && current < dayjs().startOf('day')}` trên `<DatePicker />` để cản trở chọn các ngày trước hôm nay.
  * Thêm bộ xác thực `validator`: Kiểm tra nếu hạn chót thay đổi từ giá trị ban đầu thì bắt buộc hạn chót mới không được trước phút hiện tại (`dayjs().startOf('minute')`). Việc này cho phép lưu các task overdue cũ mà không ép buộc người dùng phải đổi hạn chót nếu không sửa trường này.
* **Backend (`validation.ts`)**: Thêm `.refine()` vào `createTaskSchema` kiểm tra hạn chót lớn hơn hoặc bằng thời điểm hiện tại (trừ đi 1 phút network skew).

### B. Giờ kết thúc làm việc sau giờ bắt đầu trong Team Charter
* **Frontend (`ProjectDetail.tsx`)**: Áp dụng thuộc tính `dependencies={['workingTimeStart']}` trên trường `workingTimeEnd`. Custom validator so sánh chuỗi thời gian `'HH:mm'` của hai TimePicker, báo lỗi đỏ nếu giờ kết thúc trước hoặc trùng giờ bắt đầu.
* **Backend (`validation.ts`)**: Thêm `.refine()` so sánh chuỗi `'HH:mm'` trong `updateCharterSchema` bắt buộc `workingTimeEnd > workingTimeStart`.
