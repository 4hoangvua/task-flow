## PHẦN 1: RÀ SOÁT TƯƠNG THÍCH MOBILE (MOBILE AUDIT) & ĐỀ XUẤT 3 CHỨC NĂNG MỚI

### 1. RÀ SOÁT TƯƠNG THÍCH MOBILE (MOBILE AUDIT)

Qua khảo sát toàn bộ mã nguồn Frontend (`frontend/src/`), chúng tôi phát hiện một số điểm chưa tối ưu và có khả năng gây vỡ/tràn giao diện trên thiết bị di động như sau:

#### A. Lỗi thuộc tính size trên Drawer trong Layout chính
*   **File bị ảnh hưởng**: [DashboardLayout.tsx](file:///c:/project/task-flow/frontend/src/layouts/DashboardLayout.tsx)
*   **Chi tiết**: Drawer điều hướng cho mobile đang sử dụng `size={280}`.
*   **Vấn đề**: Trong Ant Design 5, thuộc tính `size` của `<Drawer>` chỉ nhận giá trị `"default" | "large"`. Truyền một số nguyên `280` sẽ gây ra cảnh báo TypeScript/Console và không có tác dụng.
*   **Giải pháp**: Chuyển thành `width={280}` để xác định kích thước chiều ngang chính xác của sidebar trên mobile.

#### B. Header Modal chi tiết công việc bị tràn nút bấm trên màn hình nhỏ
*   **File bị ảnh hưởng**: [TaskDetailModal.tsx](file:///c:/project/task-flow/frontend/src/components/task/TaskDetailModal.tsx)
*   **Chi tiết**: Header của Modal đang dùng `flex justify-between items-center pr-6`.
*   **Vấn đề**: Trên màn hình điện thoại hẹp, tiêu đề "Chi tiết công việc" cùng với cụm nút "Chỉnh sửa" và "Xóa" xếp hàng ngang sẽ ép dẹt nhau, gây vỡ bố cục hoặc tràn viền Modal.
*   **Giải pháp**: Nâng cấp thành `flex flex-col sm:flex-row gap-2 sm:gap-4 justify-between items-start sm:items-center pr-6`. Tiêu đề và nút bấm sẽ tự động xếp chồng dọc trên mobile và nằm ngang trên desktop.
*   **Thêm nữa**: Các bình luận dài hoặc tiêu đề bình luận cũng cần cấu hình wrap tốt (`flex-wrap gap-2`) để tránh tràn viền.

#### C. Bảng danh sách người dùng của Admin Dashboard chưa hỗ trợ Mobile
*   **File bị ảnh hưởng**: [AdminDashboard.tsx](file:///c:/project/task-flow/frontend/src/pages/AdminDashboard.tsx)
*   **Chi tiết**: Danh sách người dùng của hệ thống đang hiển thị dưới dạng `<Table>` của Ant Design trên cả Mobile và Desktop.
*   **Vấn đề**: Table chứa nhiều cột (Người dùng, Quyền, Trạng thái, Ngày tạo) sẽ bị tràn hoặc chữ bị co cụm không đọc được trên mobile.
*   **Giải pháp**: Giống như `MyTasks.tsx` và danh sách thành viên dự án, cần phát hiện kích thước màn hình bằng state `isMobile` (lắng nghe sự kiện `resize`). 
    - Nếu là Mobile: Ẩn Table và hiển thị danh sách dạng Card (User Card) sắp xếp dọc, hiển thị đầy đủ thông tin: ảnh đại diện, email, select vai trò và switch trạng thái tài khoản.
    - Nếu là Desktop: Giữ nguyên Table để tối ưu hóa không gian.

### 2. ĐỀ XUẤT 3 CHỨC NĂNG MỚI (3 NEW FEATURES RESEARCH)

Dưới đây là 3 chức năng mới được đề xuất nghiên cứu chi tiết:

#### Chức năng 1: Quản lý Công việc Con (Subtasks / Checklist)
*   **Mục tiêu**: Cho phép chia nhỏ một nhiệm vụ lớn thành nhiều đầu mục công việc nhỏ cần hoàn thành.
*   **Phạm vi nghiên cứu**: **Cả hai (Frontend & Backend)**
*   **Chi tiết Kỹ thuật**:
    1.  **Backend (Database & API)**:
        -   Thêm model `Subtask` vào `schema.prisma`:
            ```prisma
            model Subtask {
              id        String   @id @default(uuid())
              title     String
              isDone    Boolean  @default(false)
              taskId    String
              task      Task     @relation(fields: [taskId], references: [id], onDelete: Cascade)
              createdAt DateTime @default(now())
              updatedAt DateTime @updatedAt
            }
            ```
        -   Bổ sung các API endpoint:
            -   `POST /api/tasks/:taskId/subtasks` (Tạo subtask mới)
            -   `PUT /api/subtasks/:id` (Cập nhật tên hoặc trạng thái check/uncheck)
            -   `DELETE /api/subtasks/:id` (Xóa subtask)
    2.  **Frontend (UI & Logic)**:
        -   Vẽ giao diện phác thảo bằng **Stitch MCP** để hiển thị Checklist trực quan trong Modal chi tiết công việc.
        -   Thêm thanh tiến độ (Progress bar) thể hiện tỷ lệ hoàn thành (ví dụ: 3/5 công việc con đã hoàn thành - 60%).
        -   Danh sách checklist hỗ trợ thêm nhanh bằng input, bấm check để đổi trạng thái, và hover để hiện nút xóa.

#### Chức năng 2: Hệ thống Nhãn Tùy chỉnh (Custom Project Labels / Tags)
*   **Mục tiêu**: Giúp các dự án tự định nghĩa nhãn phân loại (Bug, Feature, Technical Debt, UI/UX...) kèm màu sắc nhận diện riêng, thay vì chỉ có độ ưu tiên cố định.
*   **Phạm vi nghiên cứu**: **Cả hai (Frontend & Backend)**
*   **Chi tiết Kỹ thuật**:
    1.  **Backend (Database & API)**:
        -   Thêm model `Label` và bảng quan hệ N-N trong `schema.prisma`:
            ```prisma
            model Label {
              id        String   @id @default(uuid())
              name      String
              color     String   // Mã màu HEX, ví dụ: #ef4444
              projectId String
              project   Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
              tasks     Task[]   // Quan hệ nhiều-nhiều với Task
            }
            ```
        -   Bổ sung API:
            -   CRUD Label theo từng `projectId`.
            -   API gán/gỡ nhãn khỏi Task.
    2.  **Frontend (UI & Logic)**:
        -   Thiết kế giao diện quản lý nhãn bằng **Stitch MCP** đặt trong Tab Cấu hình dự án (Settings).
        -   Cập nhật `TaskFormModal` để cho phép chọn nhiều nhãn bằng tag select của Ant Design.
        -   Hiển thị nhãn trên Kanban Card và `TaskDetailModal`.

#### Chức năng 3: Xuất báo cáo dữ liệu dự án ra CSV/Excel (Project Data Export)
*   **Mục tiêu**: Hỗ trợ Leader xuất toàn bộ danh sách công việc, trạng thái, người thực hiện, hạn chót và Timeline lịch sử hoạt động ra file Excel/CSV phục vụ mục đích lưu trữ hoặc báo cáo.
*   **Phạm vi nghiên cứu**: **Cả hai (Frontend & Backend)**
*   **Chi tiết Kỹ thuật**:
    1.  **Backend (API)**:
        -   Tạo endpoint `GET /api/projects/:id/export` (chỉ Leader/Admin được phép truy cập qua `rbacMiddleware`).
        -   Query dữ liệu từ database (Tasks, Assignees, Creators, CommentsCount, Lịch sử).
        -   Dùng thư viện chuyển đổi dữ liệu thành CSV (ví dụ: `json2csv`) và stream trực tiếp về client dưới dạng file download.
    2.  **Frontend (UI)**:
        -   Dùng **Stitch MCP** vẽ vị trí nút "Xuất báo cáo" đặt cạnh Breadcrumb dự án hoặc trong tab Overview/Settings.
        -   Khi click nút, gọi API tải file và hiển thị progress loading, lưu file về máy người dùng với tên định dạng `dự-án-name_export_YYYY-MM-DD.csv`.

#### Kế hoạch Kiểm thử & Xác thực (Cho Phần 1)
1.  **Mobile view check**: Sử dụng tính năng Responsive Emulator trong trình duyệt (Chrome DevTools) kiểm thử ở các độ phân giải:
    -   Mobile Portrait: 375x812 (iPhone X/12)
    -   Mobile Landscape: 812x375

2.  **TypeScript compile**: Kiểm tra kiểu dữ liệu sau khi sửa đổi giao diện để đảm bảo không lỗi biên dịch:
    -   Frontend: `npx tsc -b` trong thư mục `frontend/`
3.  **Thiết kế Stitch trước**: Chạy Stitch MCP để phác thảo các UI mới của 3 chức năng trước khi đề xuất triển khai code.

## PHẦN 2: PHÂN TÍCH & THIẾT KẾ TÍNH NĂNG TEAM CHARTER (QUY TẮC NHÓM)

### 1. Tổng quan nghiên cứu

#### Mục đích Team Charter (từ Tài liệu C2)
- Chuẩn hóa quy trình hoạt động nhóm
- Đảm bảo luồng thông tin trong nhóm được thông đạt
- Tạo môi trường làm việc minh bạch, hỗ trợ lẫn nhau
- Cung cấp tiêu chuẩn để nhóm tự kiểm tra, đánh giá tiến độ

#### Các thành phần chính của Team Charter (C2, mục 2.2.2)
1. **Thời gian:** Thời gian bắt đầu và kết thúc làm việc nhóm
2. **Địa điểm:** Thống nhất địa điểm làm việc cụ thể
3. **Quy luật trao đổi thông tin (Nói và Nghe):**
   - Giới hạn thời gian phát biểu
   - Nội dung vắn tắt, ngắn gọn
   - Đóng góp ý kiến mang tính xây dựng, hòa nhã, thân thiện
   - Đặt lợi ích tập thể lên trên hết
4. **Khen thưởng / Kỷ luật:** Quy định rõ ràng cho từng cá nhân
5. **Chức năng & Quyền hạn:** Quy định rõ cho từng cá nhân

#### Ai thiết lập?
- Quy tắc do **cả nhóm** thảo luận và đồng thuận
- **Trưởng nhóm (Leader)** đóng vai trò then chốt: khuyến khích thảo luận, duy trì trọng tâm, giúp nhóm ra quyết định thống nhất

---

### 2. Ý tưởng ứng dụng vào TaskFlow

#### Vị trí UI
- Thêm **tab "Quy tắc nhóm"** (Team Charter) trong trang `ProjectDetail`, ngay sau tab "Nhật ký hoạt động" và trước tab "Cấu hình dự án"
- Chỉ **LEADER** mới có quyền chỉnh sửa nội dung Team Charter
- **MEMBER** có thể xem (read-only) toàn bộ quy tắc

#### Data Model (Prisma Schema)
Thêm model `TeamCharter` liên kết 1-1 với `Project`:
```prisma
model TeamCharter {
  id              String   @id @default(uuid())
  projectId       String   @unique
  project         Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)

  workingTimeStart String? // VD: "08:00"
  workingTimeEnd   String? // VD: "17:00"
  workingDays      String? // VD: "Mon,Tue,Wed,Thu,Fri" (JSON hoặc CSV)
  workingLocation  String? // Mô tả địa điểm làm việc

  communicationRules String? // Quy luật trao đổi thông tin (rich text)
  rewardRules        String? // Quy định khen thưởng
  disciplineRules    String? // Quy định kỷ luật
  rolesDescription   String? // Mô tả chức năng & quyền hạn bổ sung

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

#### Backend API
- `GET /api/projects/:projectId/charter` — Lấy Team Charter (auth required, project member)
- `PUT /api/projects/:projectId/charter` — Tạo/Cập nhật Team Charter (Leader only)

#### Frontend Components
- Tab mới "Quy tắc nhóm" trong `ProjectDetail.tsx`
- Form chỉnh sửa với các trường: thời gian, địa điểm, quy tắc trao đổi, khen thưởng, kỷ luật
- Mode xem (read-only) cho Member, mode chỉnh sửa cho Leader
- Dùng `Input`, `TimePicker`, `Input.TextArea`, `Checkbox.Group` từ Ant Design

#### Validation (Zod)
```typescript
export const updateCharterSchema = z.object({
  workingTimeStart: z.string().optional().nullable(),
  workingTimeEnd: z.string().optional().nullable(),
  workingDays: z.string().optional().nullable(),
  workingLocation: z.string().max(500).optional().nullable(),
  communicationRules: z.string().max(2000).optional().nullable(),
  rewardRules: z.string().max(2000).optional().nullable(),
  disciplineRules: z.string().max(2000).optional().nullable(),
  rolesDescription: z.string().max(2000).optional().nullable(),
});
```
